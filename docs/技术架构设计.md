## 三、API 设计

### 3.1 通用规范

| 项目 | 规范 |
|------|------|
| 基础路径 | `/api/v1/` |
| 请求格式 | `application/json` |
| 鉴权方式 | `X-Device-Id` 请求头（客户端首次启动生成 UUID 并持久化） |
| 成功响应 | `{ "code": 0, "message": "success", "data": {...} }` |
| 错误响应 | `{ "code": <错误码>, "message": "<描述>", "data": null }` |
| 超时 | 生成接口 10s，其他接口 5s |

### 3.2 接口列表

| 编号 | 方法 | 路径 | 说明 | 核心逻辑 |
|------|------|------|------|----------|
| API-01 | POST | `/api/v1/generate` | 生成怼人话术 | 校验每日限额 → 敏感词过滤输入 → 构建Prompt → 调用AI → 过滤输出 → 持久化 → 返回 |
| API-02 | GET | `/api/v1/history` | 分页获取历史 | 按 device_id + created_at DESC 分页，返回 content_preview(前30字) |
| API-03 | DELETE | `/api/v1/history/{id}` | 删除单条历史 | 校验记录归属(device_id匹配) → 删除 → 同步清理收藏 |
| API-04 | DELETE | `/api/v1/history` | 清空全部历史 | 按 device_id 批量删除 |
| API-05 | POST | `/api/v1/favorite/{id}` | 收藏/取消收藏 | 切换 favorited 状态 + 同步维护 t_favorite 表 |
| API-06 | GET | `/api/v1/favorites` | 分页获取收藏 | 查 t_generation_record WHERE favorited=1 或 JOIN t_favorite |
| API-07 | GET | `/api/v1/usage` | 获取使用次数 | 查 Redis 当日计数 + MySQL 累计计数 |
| API-08 | POST | `/api/v1/share-image` | 生成分享图片 | 获取记录 → 渲染 Canvas 排版参数 → 生成图片 → 上传 CDN → 返回 URL |
| API-09 | GET | `/api/v1/record/{id}` | 获取单条记录详情 | 校验 device_id 归属 → 返回完整记录内容（复用 API-01 响应 data 结构）|

> **详细定义**：请求体、响应体、字段说明见 `docs/API需求.md`，本文档仅做架构层面汇总。两份文档的接口编号一一对应。

### 3.3 错误码体系

| 错误码 | 含义 | 触发条件 |
|--------|------|----------|
| 0 | 成功 | — |
| 40001 | 场景输入为空或超长 | scene 为空 或 len > 200 |
| 40002 | 风格参数无效 | style 不在枚举范围内 |
| 40003 | 记录 ID 不存在 | 查询/操作不存在的 generation_id |
| 40004 | 参数校验失败 | @Valid 校验不通过 |
| 40005 | 记录不属于当前设备 | device_id 不匹配（越权防护） |
| 42901 | 今日生成次数已达上限 | 当日计数 ≥ 50 |
| 42902 | 请求频率过高 | 设备级 QPS 超限 |
| 50001 | AI 服务异常 | DeepSeek API 超时/返回异常 |
| 50002 | 数据库异常 | MySQL 连接失败/写入失败 |
| 50003 | 图片生成失败 | Canvas 渲染/上传 CDN 异常 |
| 50004 | Redis 不可用 | Redis 连接失败（降级到 MySQL 直接读写） |

### 4.2 详细表结构

#### t_generation_record（生成记录表）

```sql
CREATE TABLE t_generation_record (
    id          VARCHAR(32)  NOT NULL PRIMARY KEY COMMENT '生成记录ID (gen_ + yyyyMMdd + 6位序号)',
    device_id   VARCHAR(64)  NOT NULL COMMENT '设备唯一标识 (客户端UUID)',
```

#### t_favorite（收藏记录表）

```sql
CREATE TABLE t_favorite (
    id             BIGINT       NOT NULL AUTO_INCREMENT PRIMARY KEY,
    device_id      VARCHAR(64)  NOT NULL COMMENT '设备唯一标识',
    generation_id  VARCHAR(32)  NOT NULL COMMENT '生成记录ID (关联 t_generation_record.id)',
```

> **双写策略**：收藏操作同时写入 `t_favorite` 表（独立记录）和更新 `t_generation_record.favorited`（冗余字段）。查询收藏列表优先走 `idx_device_favorite` 索引（单表查询），无需 JOIN。不一致时以 `t_favorite` 为准。

#### t_device（设备信息表）

```sql
CREATE TABLE t_device (
    id           BIGINT       NOT NULL AUTO_INCREMENT PRIMARY KEY,
    device_id    VARCHAR(64)  NOT NULL UNIQUE COMMENT '设备UUID (客户端生成并持久化)',
    daily_count  INT          NOT NULL DEFAULT 0 COMMENT '当日已生成次数',
    daily_date   DATE         COMMENT '计数日期 (跨天时通过此字段判断重置)',

    created_at   DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at   DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
```

> **计数策略**：Redis 作为计数主存储（INCR + TTL 48h），MySQL 作为持久化备份防止 Redis 重启丢失。每日首次请求时比对 `daily_date` 与当前日期，不一致则重置 Redis 和 MySQL 计数。

### 4.3 索引策略速查

| 表 | 索引 | 类型 | 覆盖查询 | QPS 估算 |
|----|------|------|----------|----------|
| `t_generation_record` | `idx_device_created` | 联合索引 | `WHERE device_id=? ORDER BY created_at DESC LIMIT 20` | 高 (列表页) |
| `t_generation_record` | `idx_device_favorite` | 联合索引 | `WHERE device_id=? AND favorited=1` | 中 (收藏页) |
| `t_favorite` | `uk_device_gen` | 唯一索引 | 收藏/取消时的去重校验 | 中 |
| `t_favorite` | `idx_device` | 联合索引 | 收藏列表按时间分页 (备用方案) | 低 |
| `t_device` | `idx_device_date` | 联合索引 | `WHERE device_id=? AND daily_date=?` | 高 (每次生成) |
| `t_sensitive_word` | `uk_word` | 唯一索引 | 敏感词精确匹配 | 内存 (Caffeine) |

### 4.4 Redis 数据结构

| Key 模式 | 类型 | 示例 | 用途 | TTL |
|----------|------|------|------|-----|
| `daily:{device_id}:{yyyyMMdd}` | string | `daily:abc123:20260527` | 当日生成次数计数器 (INCR) | 48h |
| `rate:{device_id}:{window}` | string | `rate:abc123:1716800000` | 设备级滑动窗口限流 (1s窗口) | 2s |
| `rate:ip:{ip}:{window}` | string | `rate:ip:10.0.0.1:1716800000` | IP级别限流兜底 | 2s |
| `gen:{id}` | hash | `gen:gen_20260527_000001` | 单条生成记录缓存 (热点数据) | 30min |
| `sensitive:words:level1` | set | — | 级别1敏感词集合 (拦截级，启动加载) | 持久 |
| `sensitive:words:level2` | set | — | 级别2敏感词集合 (替换级，启动加载) | 持久 |

> **命名规范**：`业务域:{标识}:{维度}` 三段式，冒号分隔。计数器类用 string + INCR，集合类用 set，缓存类用 hash。

## 五、关键设计决策

以下记录架构层面的关键权衡及选择理由，供后续迭代参考。

### 5.1 收藏双写 vs 单表方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| 单表方案 | 只用 `t_generation_record.favorited` 字段 | 简单，无一致性风险 | 列表查询需 `WHERE favorited=1`，无法记录收藏时间线 |
| **双写方案（选用）** | `t_favorite` 独立表 + `favorited` 冗余字段 | 收藏列表走 `idx_device_favorite` 单表查询，性能好 | 需维护双写一致性 |

**决策理由**：V1 阶段收藏查询频率远高于写入频率，冗余字段换读取性能是合理的。不一致时以 `t_favorite` 表为准，定时任务可做最终一致性校验。

### 5.2 Redis 主计数 + MySQL 备份 vs Redis-Only

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| Redis-Only | 每日计数仅存 Redis | 极简，一读一写 | Redis 重启/清空计数丢失 |
| **Redis+MySQL（选用）** | Redis INCR 作为主力，MySQL 作为持久化备份 | 高可用，计数不丢 | 多一次写操作 |

**决策理由**：每日限额是核心限流手段，丢失计数会导致用户超限使用，对 AI API 成本有直接影响。写 MySQL 的额外开销在生成场景(>1s)中可忽略。

### 5.3 Taro（React）vs 微信原生

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| 微信原生 | 使用 WXML/WXSS/JS 原生开发 | 包体积最小，启动性能最优 | 无法多端复用，语法封闭，TypeScript 支持有限，不支持 npm 生态直接使用 |
| **Taro + React（选用）** | Taro 4.x + React + TypeScript | React 生态复用，Ant Design Mobile 组件库，TypeScript 类型安全，后续可编译到 H5/支付宝/抖音等平台 | 编译层有少量性能开销，包体积略大 |

**决策理由**：虽然 V1 页面不多，但 Taro + React 提供 TypeScript 类型安全保障，Ant Design Mobile 提供成熟的 UI 组件库减少样式开发量，且 Taro 4.x 编译优化已使包体积和启动速度接近原生。React 生态的 hooks、状态管理等模式也利于代码组织和后续迭代。若后续需扩展到其他平台，无需重新开发。

### 5.4 敏感词加载策略：启动全量 + 双层缓存

```
启动时: MySQL → Redis Set (持久) + Caffeine 本地缓存 (内存)
运行时: ContentFilter → Caffeine (L1) → Redis Set (L2, 兜底)
```

**决策理由**：敏感词库数据量小（预估 < 5000 条），全量加载到内存无压力。Caffeine 作为 L1 缓存提供纳秒级匹配，Redis Set 作为 L2 提供多实例共享和管理后台动态更新的同步能力。

### 5.5 AI SDK 选型：OpenAI SDK 连接 DeepSeek

**决策理由**：DeepSeek API 完全兼容 OpenAI 协议，使用 `openai-java` SDK 仅需修改 `baseUrl` 和 `apiKey` 即可接入。后续如需切换模型供应商（如通义千问、GLM），只要供应商兼容 OpenAI 协议即可零代码切换。

```
你是一个"怼人话术生成器"，你帮助用户用幽默、高情商的方式表达情绪。
```

## 九、安全设计

### 9.1 三层安全防线

```
用户输入
    │
    ▼
┌──────────────────────┐
│ 第一层：输入过滤      │  Controller @Valid 参数校验 + ContentFilter 敏感词匹配
│                      │  命中 → 40001 / 拒答兜底文案
└──────────┬───────────┘
           │ 通过
           ▼
┌──────────────────────┐
│ 第二层：Prompt 约束   │  System Prompt 内嵌安全原则，限缩 AI 行为边界
│                      │  异常 → 50001
└──────────┬───────────┘
           │ AI 返回
           ▼
┌──────────────────────┐
│ 第三层：输出过滤      │  ContentFilter 对 AI 返回结果二次检查
│                      │  命中 Level1 → 替换为拒答兜底文案
│                      │  命中 Level2 → 替换为 ***
└──────────┬───────────┘
           │ 通过
           ▼
      返回用户
```

### 9.2 限流矩阵

| 层级 | 实现 | 维度 | 阈值 | 超限响应 |
|------|------|------|------|----------|
| Nginx | `limit_req` | 全局 IP | 100 r/s | 503 |
| Nginx | `limit_req` | 设备 ID | 10 r/s | 503 |
| SpringBoot | `RateLimitInterceptor` + Redis | 设备 ID (滑动窗口) | 精准控制 | 42902 |
| SpringBoot | Redis INCR | 设备每日生成次数 | 50 次/天 | 42901 |
| 小程序端 | 按钮 disabled + 防抖 | 点击频率 | 1次/2s | 按钮置灰 |

### 9.3 敏感词双层缓存架构

```
启动加载:
  MySQL t_sensitive_word ──► Redis Set (L2, 持久) ──► Caffeine (L1, 内存)

运行时匹配:
  ContentFilter.match(word)
    ├── L1: Caffeine.get(word) → 命中直接返回 (ns级)
    └── L2: Redis SISMEMBER → 未命中则回源 (us级) → 回填 L1

管理后台更新敏感词后:
  MySQL INSERT → 发 Redis PUBLISH → 各实例监听到 → 刷新 Caffeine
```

### 10.4 请求封装

```typescript
// utils/api.ts
import Taro from '@tarojs/taro';
import { getDeviceId } from './device';

const BASE_URL = 'https://api.aiquarrel.com/api/v1';

function request<T = any>(path: string, options: Taro.request.Option = {}): Promise<T> {
  return new Promise((resolve, reject) => {
    Taro.request({
      url: BASE_URL + path,
      method: options.method || 'GET',
      data: options.data,
      header: {
        'Content-Type': 'application/json',
        'X-Device-Id': getDeviceId()
      },
      timeout: options.timeout || 15000,
      success(res) {
        if (res.data.code === 0) resolve(res.data.data as T);
        else reject(res.data);
      },
      fail() {
        reject({ code: -1, message: '网络不给力，请检查网络后重试' });
      }
    });
  });
}
```
